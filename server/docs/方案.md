**后端方案书**


1. 目标检测服务负责“找出每个榴莲 + 编号”
2. 多模态/大语言模型负责“基于已编号图片 + 每个目标裁剪图做评分和解释”


---

**一、项目目标**

输入一张榴莲货架图片，后端完成：

1. 识别画面中的所有榴莲
2. 给每个榴莲生成稳定编号（A、B、C...）
3. 生成带编号的标注图
4. 调用大模型逐个评分并给出解释
5. 返回推荐结果、排序结果、可分享图片及结构化数据

输出效果对应你给的三张图：
- 图一 -> 原始上传图
- 图二 -> 后端检测并编号后的图
- 图三 -> 后端汇总大模型评分、推荐最佳榴莲与说明

---

**二、后端总体架构**

建议拆成 5 个模块：

1. API 网关层  
接收图片上传、任务创建、结果查询、分享图生成请求

2. 识别与标注服务  
负责榴莲检测、去重、排序编号、生成标注图

3. AI 评分服务  
负责组织提示词、调用多模态模型、解析结构化结果、生成推荐排序

4. 任务与存储服务  
负责异步任务、图片存储、结果缓存、日志审计

5. 运维与监控模块  
负责限流、失败重试、耗时监控、模型调用成本统计

推荐采用：
- `Node.js + NestJS` 做主后端
- `Python` 做视觉识别微服务
- `MySQL/PostgreSQL` 存任务和结果
- `Redis` 做队列与缓存
- `COS/OSS/S3` 存原图、标注图、分享图

---

**三、核心处理流程**

**1. 用户上传图片**
- 小程序上传原图到对象存储
- 后端创建 `analysis_task`

**2. 榴莲检测与编号**
- 检测服务识别图中所有榴莲边界框
- 按“从左到右、从上到下”排序
- 依次编号 A、B、C...
- 生成标注图

**3. 构造 AI 分析输入**
- 输入给大模型两类材料：
  - 带编号的整图
  - 每个榴莲的局部裁剪图
- 同时附上规则化提示词：
  - 成熟度
  - 果形饱满度
  - 刺的密度与粗细
  - 果柄与表皮状态
  - 裂口/黑斑/损伤风险
  - 推荐购买优先级

**4. 大模型输出结构化结果**
要求模型返回 JSON：
- `label`
- `score`
- `summary`
- `reasons`
- `risks`
- `buy_priority`

**5. 结果后处理**
- 校验 JSON
- 分数归一化
- 选出 Top1 / Top3
- 生成最终展示结果与分享图

---

**四、关键技术方案**

**1. 榴莲识别方案**

建议优先选传统 CV/检测模型，不要让 LLM 负责检测。

可选方案：
1. `YOLOv8 / YOLO11` 自训练检测模型
2. 若数据量小，先用通用目标检测模型做迁移学习
3. MVP 阶段也可先人工收集几十到几百张图片做小样本训练

识别服务输出：
```json
[
  { "id": "A", "bbox": [x, y, w, h], "confidence": 0.95 },
  { "id": "B", "bbox": [x, y, w, h], "confidence": 0.93 }
]
```

排序规则必须固定，否则同一张图多次识别会出现编号漂移。建议规则：
- 先按中心点 `y` 分层
- 同层内按 `x` 升序
- 层间按 `y` 升序

这是业务稳定性的关键。

**2. 大模型评分方案**

建议采用“视觉输入 + 强约束 JSON 输出”。

模型职责：
- 不负责找榴莲
- 只负责看编号图和裁剪图，对已知目标做解释性评分

评分维度建议：
- 外观成熟度：25
- 果形饱满度：20
- 刺形与壳薄肉厚特征：20
- 新鲜度与损伤风险：20
- 综合购买建议：15

总分 100。

输出示例：
```json
{
  "label": "G",
  "score": 88,
  "summary": "成熟度较好，果形饱满，优先推荐",
  "reasons": [
    "颜色黄绿适中",
    "果形圆润饱满",
    "果刺偏钝且间距适中"
  ],
  "risks": [
    "未见明显开裂，建议尽快食用"
  ],
  "buy_priority": 1
}
```

**3. 提示词工程**

提示词必须明确三件事：
1. 只分析已编号榴莲，不新增编号
2. 必须按统一维度打分
3. 必须返回 JSON，不能输出自由文本

同时要加“保守规则”：
- 看不清时降低置信
- 不允许臆测果肉实际口感
- 只能基于图像外观做判断

**五、后端接口设计**

建议首版提供以下 API。

**1. 创建分析任务**
`POST /api/v1/durians/analyze`

请求：
- 图片 URL 或上传文件 ID

返回：
- `taskId`
- `status`

**2. 查询任务状态**
`GET /api/v1/durians/tasks/{taskId}`

返回：
- `PENDING | DETECTING | SCORING | DONE | FAILED`

**3. 获取分析结果**
`GET /api/v1/durians/tasks/{taskId}/result`

返回：
- 原图 URL
- 标注图 URL
- 推荐榴莲
- 全部评分列表
- 文案说明
- 分享图 URL

**4. 重新分析**
`POST /api/v1/durians/tasks/{taskId}/retry`

**5. 获取分享图**
`GET /api/v1/durians/tasks/{taskId}/share-card`

建议至少 3 张核心表。

**1. `analysis_task`**

- `source_image_url`
- `annotated_image_url`
- `status`

- `created_at`

**2. `analysis_item`**

**3. `analysis_model_log`**

- `model_name`
- `prompt_version`
- `request_token`
- `response_token`
- `latency_ms`
- `raw_response`

这张日志表很重要，用于后续优化提示词和排查误判。

---

**七、异步任务设计**

这类流程不适合同步接口直出，建议异步化：

1. 上传成功后写入任务表
2. 投递到队列
3. 检测服务消费任务
4. 检测完成后投递评分任务
5. 大模型评分完成后汇总结果
6. 前端轮询或订阅状态

推荐：
- `BullMQ + Redis`
- 每步都支持重试
- 失败任务可人工回放

---

**八、分享图生成方案**

分享图由后端统一生成，避免前端排版不一致。

内容包含：
- 标注图
- 推荐编号（如 G）
- 推荐文案
- Top3 排名
- 总分

可用：
- `node-canvas`
- 或服务端 HTML 模板转图片

---

**九、稳定性与风控**

**1. 质量控制**
- 检测数量过少或过多时报警
- 编号冲突时直接失败
- 大模型返回非 JSON 时自动重试一次

**2. 成本控制**
- 先检测再裁剪，减少传给模型的无效区域
- 图片压缩到合理尺寸
- 同一任务结果缓存，避免重复调用模型

**3. 可解释性**
- 所有推荐结果都保留“评分理由”
- 保存原始模型输出，便于复盘

**4. 版本化**
- 检测模型版本
- 提示词版本
- 评分规则版本
都要落库，否则后续结果无法追溯

**十、实施阶段建议**

**Phase 1：MVP**
目标：先跑通完整链路
- 图片上传
- 榴莲检测与编号
- 标注图生成
- 大模型评分
- Top1 推荐返回

**Phase 2：增强版**
目标：提升准确率和可运营性
- 多图分析
- Top3 推荐
- 分享图生成
- 失败重试与监控

**Phase 3：产品化**
目标：可稳定上线
- 用户历史记录
- 模型版本管理
- 成本统计
- 运营后台查看任务与错误日志

---

**十一、推荐技术选型**

如果你要尽快落地，我建议：

- 主后端：`NestJS`
- 视觉服务：`Python + FastAPI + YOLO`
- 队列：`Redis + BullMQ`
- 数据库：`PostgreSQL`
- 文件存储：`COS/OSS/S3`
- AI 模型调用：支持图片输入并可结构化输出的多模态模型
- 部署：`Docker + ECS/云主机`

原因很直接：
- Node 适合业务 API、队列、分享图和工程组织
- Python 更适合视觉模型推理
- 两者拆分后，后续替换检测模型或大模型都更容易

---

**十二、主要风险**

1. 仅凭单张外观图，无法真正判断榴莲“口感”和“开果成熟度”，只能做外观推荐
2. 榴莲遮挡严重时，检测与编号稳定性会下降
3. 大模型解释能力强，但评分一致性可能波动，必须加结构化约束和版本管理
4. 如果没有自有数据集，检测准确率初期会受限
